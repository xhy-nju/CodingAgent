# CodingAgent 项目个人反思

这次期末项目中，我选择实现一个 Coding Agent Harness，并把治理护栏、反馈闭环、记忆和工具分发作为主要贡献。最初我对 coding agent 的理解更接近“让大模型调用几个工具完成任务”，但随着设计推进，我逐渐认识到，真正困难的不是让模型生成代码，而是约束它的行动、验证结果，并在失败后形成可追踪的下一步。因此，我把治理护栏和 HITL 审批确定为第一主贡献，把反馈闭环作为第二重点，再用确定性记忆和工具分发支撑运行过程。安全性不依赖提示词声明，而由代码规则、状态机和测试证明。

项目采用了 Spec-Driven 的开发方式。开始时，我觉得先写 `SPEC.md` 和详细 `PLAN.md` 会拖慢进度，因为未实现的内容只能依靠预判。但冷启动验证改变了我的看法。我把规约和计划交给一个全新的 opencode 会话，让它尝试实现 Task 4。它虽然理解了 Guardrail 和审批的大方向，却暴露出几个具体问题：`run_command` 白名单与“审批后执行”的语义冲突，脱敏样例和正则规则不一致，而且 Task 4 的前置依赖不够明确。如果直接进入实现，它们很可能在后期变成接口冲突。经过验证，我补充了前置依赖、审批规则和冷启动范围，直观感受到上下文质量会决定 AI 实现的稳定性。

在核心模块开发中，TDD 对我帮助很大。Action Parser、Guardrail、Approval、SQLite Store、Memory 和 AgentLoop 都先通过失败测试明确接口，再补上最小实现。这样的红绿循环有时显得机械，却能防止 AI 在一次修改中扩展过多范围。例如，HITL 不只是把状态从 `pending` 改成 `approved`，还必须保证同一个批准动作只能执行一次，并在执行后恢复原来的运行。通过 exactly-once 测试，这项要求从文字描述变成了可以重复验证的行为。反馈闭环的 Mock 演示也让我认识到，模型“声称已经修复”没有意义，只有工具真实返回的 pytest 结果才能成为下一轮输入。Mock LLM 因而不是对真实模型的替代品，而是隔离网络波动和模型随机性、验证 Harness 机制的实验装置。

真实 LLM 的接入让我进一步理解了工程边界。项目使用 OpenAI-compatible 接口和 `glm-5.2`，但我没有让真实模型成为默认验收路径，而是保留 Mock/Real 双模式。真实模式必须经过显式开关和管理员认证，并与 Mock 共用 Action Protocol、Guardrail、工具、反馈、Memory 和审计链路。凭据处理尤其让我意识到，“API Key 没有提交到 Git”只是最基本的要求。密钥还可能通过日志、SSE、SQLite、错误信息或模型提示词间接泄露。因此，项目最终统一了 Docker Secret、环境变量和 Keyring 的凭据来源，并在持久化和输出前执行脱敏。这个过程使我对安全的理解从“隐藏一个字符串”转向“检查数据经过的每个边界”。

前端和部署阶段说明单元测试通过并不等于产品路径可用。WebUI 曾经所有按钮都返回 500，根因不是 AgentLoop，而是 API 把演示工作区创建在无权限的 Windows 临时目录中。修复后，我增加了工作区位置回归测试，并重新从浏览器完成一键演示。Docker 分发也带来类似经验：`.dockerignore`、静态资源构建、Compose 环境变量、数据卷、健康检查和管理员密码缺一不可。最终同时验证 CLI、API、WebUI、Docker 和真实模型探针，比只运行一次 `pytest` 更接近真实交付。

Superpowers 工作流在这个项目中既有价值，也有明显成本。Brainstorming 和 Writing Plans 帮助我提前锁定主贡献，Systematic Debugging 避免了在 500 错误上盲目改代码，Verification Before Completion 则要求结论必须有命令和输出支持。另一方面，多轮审查、子代理复核和频繁的阶段确认也显著降低了推进速度。尤其在边界已经清晰的后期任务中，继续重复全面复核带来的收益有限。我后来明确调整为“先完成一个相对完整的部分，再集中测试，出现问题后修正”，开发节奏才恢复正常。这让我认识到，流程本身也需要治理。严格流程适合安全、凭据和跨模块状态机等高风险部分，而简单文档或局部界面不应使用相同强度的审查。

如果重新完成一次，我会保留 Spec-first、冷启动验证和核心机制的 TDD，但会重新划分任务颗粒度。领域模型、协议解析和 Guardrail 适合小任务逐步推进；WebUI 应拆成 API Client、Dashboard/Run Detail、管理视图三个独立任务；Docker 镜像与服务器部署文档也应该分开。我还会更早定义“自动测试证据”和“人工演示证据”的边界，避免在后期反复确认哪些功能必须一键展示。对于子代理，我会只分派接口稳定、文件边界明确的任务，并由主流程统一验证，而不是为了形式上的并行而增加沟通成本。

这次项目最大的收获，是我不再把 coding agent 看成一个更长的 Prompt。模型只是候选动作的生成者，Harness 才负责决定动作是否可信、是否允许执行、结果是否成功以及经验是否值得保留。人类也不是在最后点击一次确认，而是要决定范围、风险和开发节奏，并对最终交付负责。项目仍然只是单管理员、单机 SQLite 的课程系统，真实模型行为和公网部署也受外部环境影响，但它已经完整展示了一个 Agent 从决策到治理、执行、反馈和审计的基本闭环。相比单纯完成更多功能，我认为这种可解释、可测试和可复现的工程边界更有价值。

> AI 使用说明：本项目使用 AI 辅助梳理规约、生成和修改部分实现与测试、分析调试信息、整理过程证据并润色反思文本。项目方向、技术选型、访问边界、流程调整、功能验收和最终观点由我确认并负责。
